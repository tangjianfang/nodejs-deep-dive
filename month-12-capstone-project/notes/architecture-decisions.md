# 架构决策记录（ADR）

## 1. 什么是 ADR

**ADR (Architecture Decision Record)** 是一种轻量级文档，用于记录一个重要的架构决策
——包括决策的背景、考虑的选项、最终选择以及结果。

> "代码记录了你做了什么，ADR 记录了你为什么这样做。"

**核心价值：**
- 新成员快速理解历史决策原因
- 避免"我们当时为什么这样做？"的困惑
- 定期回顾决策是否仍然有效

## 2. ADR 标准模板

```markdown
# ADR-{编号}: {标题}

**日期**: YYYY-MM-DD
**状态**: 草案 | 已接受 | 已废弃 | 被取代（被 ADR-XX 取代）
**决策者**: @alice @bob

## 背景

{描述为什么需要做这个决策，当前面临的问题或机会}

## 考虑的选项

### 选项 A: {名称}
**优点**: ...
**缺点**: ...

### 选项 B: {名称}
**优点**: ...
**缺点**: ...

### 选项 C: {名称}
**优点**: ...
**缺点**: ...

## 决策

我们选择 **选项 B**，因为 {核心理由}。

## 后果

**正面影响**:
- {好处1}
- {好处2}

**负面影响/权衡**:
- {代价1}，我们计划通过 {方案} 缓解
- {代价2}，可以接受

## 参考资料

- {链接或资料}
```

## 3. 本仓库 12 个月的核心 ADR

### ADR-001: 选择 Node.js 作为主要学习平台

**日期**: Month-01  
**状态**: 已接受

**背景**: 需要选择一门运行时环境进行深度学习。

**考虑选项**:
- Node.js — JavaScript 运行时，事件驱动，单线程
- Go — 编译型，goroutine 并发，静态类型
- Python — 解释型，GIL 限制，数据科学友好

**决策**: 选择 **Node.js**

**理由**:
1. JavaScript 是全栈最通用的语言
2. 事件循环模型是深度理解异步编程的最好载体
3. npm 生态最庞大
4. 相同的语言可以在前端/后端/工具链复用

---

### ADR-002: 优先使用 Node.js 内置模块

**日期**: Month-01  
**状态**: 已接受

**背景**: 演示代码是否应大量引入第三方库（如 axios、lodash）？

**决策**: **优先使用内置模块**（`http`、`crypto`、`stream`、`worker_threads`…），第三方库只在必要时引入并加 `// npm install xxx` 注释。

**理由**: 内置模块理解更深，无版本兼容问题，且所有 Node.js 文档都是权威参考。

---

### ADR-003: 异步方案选择 async/await 而非 callback

**日期**: Month-02  
**状态**: 已接受

**背景**: Node.js 历史上有 callback、Promise、async/await 三种异步模式。

**决策**: 主要代码使用 **async/await**，提供 Promise 降级版本，callback 仅用于解释历史原因。

**后果**: 代码更易读，但需要理解 async/await 在底层仍是 Promise/Generator 组合。

---

### ADR-004: 流处理选择 Stream Pipeline API

**日期**: Month-03  
**状态**: 已接受

**背景**: `stream.pipe()` vs `stream.pipeline()` vs `for await...of`

**决策**: 使用 `stream.pipeline()` + `promisify`，原因：
- 自动处理错误传播和清理（`pipe()` 不会）
- 比 `for await...of` 更适合多步骤转换链

---

### ADR-005: 认证方案选择 JWT RS256

**日期**: Month-07  
**状态**: 已接受

**背景**: HS256 vs RS256 签名算法选择

**考虑选项**:
- HS256 (HMAC-SHA256) — 对称，服务端共享密钥
- RS256 (RSA-SHA256) — 非对称，私钥签名/公钥验证
- ES256 (ECDSA) — 非对称，更短密钥

**决策**: **RS256**

**理由**: 微服务场景下，多个服务可以用公钥验证 JWT，而不需要共享私钥。私钥只存在于 Auth Service，降低密钥泄漏风险。

---

### ADR-006: 缓存模式选择 Cache-Aside

**日期**: Month-09  
**状态**: 已接受

**背景**: Cache-Aside vs Write-Through vs Write-Behind

**决策**: **Cache-Aside（懒加载）**

**理由**: 最灵活，应用程序控制缓存生命周期，适合读多写少场景。代价是首次读取有缓存穿透（用 Lock / Singleflight 缓解）。

---

### ADR-007: 微服务通信选择 HTTP/gRPC 混合

**日期**: Month-10  
**状态**: 已接受

**背景**: 微服务间通信协议选择

**决策**:
- **内部服务间同步调用**: gRPC（强类型，高性能，支持流式）
- **外部客户端 API**: REST/HTTP（浏览器兼容，易调试）
- **异步解耦**: 消息队列（BullMQ/Kafka）

## 4. 如何在团队中推行 ADR

```
推荐工作流:

1. 识别重大决策（影响 > 1个月的技术方向）
2. 创建 ADR 草案，发起讨论
3. 团队 Review，记录反对意见
4. 接受并合并到代码仓库（docs/adr/ 目录）
5. 代码里通过注释引用 ADR 编号

// 使用 Fan-out on Write 策略，详见 ADR-012
async function publishTweet(userId, content) { ... }
```

**文件组织建议：**

```
docs/
  adr/
    0001-use-nodejs.md
    0002-use-builtin-modules.md
    0003-async-await.md
    README.md          ← 所有 ADR 的索引
```

## 5. 常见架构权衡速查

| 权衡           | 左边                  | 右边                  |
| -------------- | --------------------- | --------------------- |
| 一致性 vs 可用性 | 强一致（牺牲可用性）  | 最终一致（牺牲强一致）|
| 读性能 vs 写性能 | 写时扩散（读快）      | 读时聚合（写快）      |
| 简单 vs 灵活   | 单体架构              | 微服务架构            |
| 规范 vs 灵活   | GraphQL（强类型）     | REST（宽松）          |
| 开发速度 vs 维护性 | 快速原型           | 测试覆盖+文档         |
| 延迟 vs 吞吐量 | 低延迟（单连接）      | 高吞吐（批量处理）    |
